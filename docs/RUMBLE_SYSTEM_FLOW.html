Rumble Flow Map

Underground Claw Fights: target state machine and current alignment

This map turns the current codebase into a visual workflow reference. It shows the ideal stage flow, who owns each part of the system, and where the current implementation still disagrees with the intended product. The target side is the intended source of truth, including both desired behavior and accepted operating constraints that are already true today. It also captures the current access model: any supported Solana wallet can bet, the Seeker APK is optional, and trusted bot onboarding now runs through wallet signatures, Seeker Genesis checks, and allowlisting.

System Ownership

source of truth user-facing stage alignment gap

Railway Worker

Owns the queue manager, orchestrator, stage transitions, on-chain create/start/finalize flow, and mutation authority.

Vercel App + API

Serves clawfights.xyz, public UI, and public API routes. It reads shared state, but it should not own lifecycle mutation in production.

Supabase

Durable memory for active rumbles, turn logs, bet records, payout records, claim state, and betting-ready markers.

Mainnet

Betting readiness, live betting, fee collection, claimability, vault balances, and bettor payout truth.

Trust Layer

Signed wallet proof, Seeker Genesis checks, and wallet allowlisting decide who auto-approves into the live fighter path.

Devnet / ER

Combat turn state, commit/reveal windows, duel resolution, and combat pacing.

Web + Mobile

Render and wallet UX only. They should never invent stage readiness or fee math that Railway, Vercel status, and chain have not confirmed.

Access + Trust Paths

How bettors, bots, and wallet approval fit into the live system.

Bettors

Betting is not Seeker-gated. Any supported Solana wallet can use clawfights.xyz to watch live fights, place bets, and claim payouts.

Seeker APK

The APK is an optional mobile-native path for the same live system. It is a preferred UX for Seeker users, not a requirement for betting.

Fighter Registration

Fighter creation now requires signed wallet proof. A raw wallet address alone is no longer enough to register a live-ready fighter.

Auto-Approval

Fast approval comes from env allowlist, manual wallet allowlist, or verified Seeker Genesis ownership. Everything else stays pending for review.

Seeker Bots

@SeekerClaw is the preferred Seeker-native agent suggestion. Bots can load skill.md as the operating guide for registration, queueing, moves, betting, and claims.

Non-Seeker Bots

Non-Seeker bots can still join. If they do not auto-approve, they should ask @ble77_ed or @ClawFights to allowlist the wallet.

Canonical Stage Flow

Target on the left. Current implementation snapshot on the right.
1
Queue
Target: 12 start / 16 cap / 10s lock

Target

  • One active rumble runs at a time.
  • Fighters enter queue continuously.
  • When queue reaches 12, start a 10-second lock timer.
  • During those 10 seconds, keep accepting fighters up to 16.
  • Start the rumble when the timer ends or the queue hits 16.

Current

  • Minimum start is 12.
  • Default full bracket is also 12 unless env overrides it to 16.
  • Default lock countdown is 30 seconds.
  • Only one active slot exists in runtime.
default cap mismatch default countdown mismatch single-slot runtime
2
Betting Arming
Internal only

Target

  • This is a real backend stage, but it stays hidden in product UX.
  • No public timer yet.
  • No public bet button yet.
  • Mainnet readiness is the gate before betting becomes public.
  • Wait until mainnet rumble is readable and betting can actually be prepared on-chain.
  • Users should never see a fake open betting stage here.

Current

  • Queue manager already has this concept: betting exists before the public deadline is armed.
  • Status and UI now fail closed instead of showing stale live betting.
  • This stage should still be treated as hidden/internal in product UX.
logic exists UX should stay invisible
3
Betting Live
Target: exactly 60s

Target

  • This stage belongs to the single active rumble only.
  • Public timer starts only when on-chain betting is definitely live.
  • Bet button works immediately.
  • No "betting not initialized" or "slot unavailable" while timer is running.
  • Users can place bets until the minute ends.
  • Each bet takes 1% platform + 1% fighter sponsorship upfront, with 98% entering the pool.
  • No other hidden upfront fee path should exist.

Current

  • Code default is 60s, but checked-in app env currently sets 120s.
  • Queue manager also enforces a minimum visible betting window, default 45s.
  • Mainnet is the money-side readiness source.
  • On-chain fee split is now 1% treasury, 1% fighter sponsorship, 98% net pool.
current env says 120s mainnet-only readiness 2% upfront total
4
Fight Commit
Target: 30 slots ~= 12s for now

Target

  • Commit window is 30 slots, about 12 seconds at the current slot estimate.
  • Fighters/agents choose moves in a clearly defined commit window.
  • The phase must be visible and predictable.
  • Late or missing agent input should fall back cleanly.
  • When only 2 fighters remain, the very next duel becomes sudden death.

Current

  • Commit window is 30 slots.
  • At the current 400ms slot estimate, that is about 12 seconds.
  • Internal agent timeout is 3500ms before fallback logic kicks in.
  • Final-duel sudden death now forces damage: real hits get +20 and double-whiffs chip both fighters for 20.
  • House bots now react to recent dodges/strikes and push harder to finish the last duel.
timed window exists agent timeout is shorter than turn window final duel converges
5
Fight Reveal
Target: 30 slots ~= 12s for now

Target

  • Reveal window is 30 slots, about 12 seconds at the current slot estimate.
  • Moves are revealed and commentary reacts to what actually happened.
  • This phase should be legible to spectators.
  • Next commit starts only after reveal resolves.
  • The next turn should not make the current reveal unreadable.

Current

  • Reveal window is also 30 slots, about 12 seconds.
  • Next commit does not currently start during reveal.
  • Turn resolves first, then the next turn opens.
non-overlapping phases
6
Final Hold
Target: 5s hard hold minimum

Target

  • The last resolved move must stay visible before payout.
  • Winner and finishing blow must be readable.
  • Use a 5-second minimum hard hold unless the product chooses a longer value.
  • This should be guaranteed on every client, every time.

Current

  • Web client already holds the final combat turn for 5 seconds.
  • Mobile does not have the same dedicated final-turn hold path.
  • This is still one of the major alignment gaps.
web only not guaranteed on mobile
7
Payout
Display ~= 60s, claims stay open

Target

  • Keep the payout/result view stable for about 60 seconds.
  • Only first-place bettors are paid.
  • Winners receive their winning stake back plus their pro-rata share of the losers pool.
  • On-chain claims stay available until claimed.
  • The 3% losers-pool treasury cut is extracted once when result is finalized.
  • That 3% comes only from losers pool, not from winning capital.
  • Users can claim now or later.
  • Fighters get paid out cleanly.
  • The stage should feel stable, not rushed.

Current

  • Checked-in app env currently sets payout display to 60 seconds.
  • 24h now acts as a completion buffer, not a claim expiry.
  • Only first-place bettors are eligible in the winner-takes-all model.
  • Winner-takes-all treasury cut is moved once at result finalization.
  • Payout state is already durable through Supabase + chain.
durable claims 3% extracted once 60s display configured locally

Top Alignment Gaps

What still needs to become explicit in code + Railway env

Queue spec is not locked

If Railway does not set FIGHTERS_PER_RUMBLE=16 and RUMBLE_QUEUE_LOCK_COUNTDOWN_MS=10000, the live queue behavior still does not match the intended 12-to-start / 16-cap / 10-second flow.

Betting duration mismatch

Product target says 60 seconds. Checked-in app env says 120 seconds. That needs one canonical value.

Final move is not guaranteed everywhere

Web tries to hold the last turn. Mobile still needs the same hard final-turn hold before payout replaces the fight view.

Spec still lives in multiple places

The chosen flow now exists in this map, but runtime behavior can still drift if Railway env values and code constants are not pinned to the same numbers.
