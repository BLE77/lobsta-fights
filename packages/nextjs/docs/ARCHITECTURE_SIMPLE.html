Current Live Architecture

Claw Fights is a hybrid system.

Real money lives on mainnet. Fast combat runs on MagicBlock ER. Randomness comes from MagicBlock VRF. Railway runs the game worker, and Vercel serves the app.

Short Version

Money
Mainnet betting + mainnet claims
Combat
Devnet rumble state + MagicBlock ER
Randomness
VRF for matchups and Ichor Shower
Ops
Railway worker + Vercel UI/API + Supabase mirror

What Runs Where

Mainnet
Bets and claims
Real wager flow, wallet signing, bettor state, claim flow.
Devnet + ER
Combat execution
Rumble combat state delegates into MagicBlock ER for fast turns.
VRF
Fair randomness
Matchup seed before turn 1 and Ichor Shower random event.
Supabase
Status mirror
Queue, turn logs, payout rows, UI support, public status history.

End-to-End Flow

  • User opens `/rumble` on Vercel.
  • Wallet signs a mainnet bet transaction.
  • Railway closes betting and starts combat.
  • Combat state is delegated to MagicBlock ER.
  • Railway requests matchup VRF before turn 1.
  • Turns resolve on the combat side.
  • Railway finalizes the result.
  • Railway requests Ichor Shower VRF.
  • Supabase stores the result and Vercel shows it.

Simple Mental Model

  • Money side: mainnet
  • Fight side: devnet + MagicBlock ER
  • Who fights who: VRF matchup seed
  • Lottery / shower: VRF shower request
  • Game operator: Railway worker
  • Website + API: Vercel

Important Caveat

  • This is not a single-chain-only production design yet.
  • Betting is on mainnet.
  • Combat is on devnet + ER.
  • Scheduling, bots, and some product behavior are still off-chain.
Correct description: the game uses on-chain betting, on-chain ER combat, and on-chain VRF, but the full product is still a hybrid web app.
